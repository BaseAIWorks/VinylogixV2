The three critical items, up close

Where it breaks, and what breaking looks like

01 Critical · write contention

Every order increments the same document

src/services/server-order-service.ts:191–200

At order creation time — in each of the Stripe, PayPal, and manual flows — a Firestore transaction reads the distributor's orderCounter field, increments it, and writes it back. Firestore enforces a hard ceiling of approximately one write per second against any single document.

Under current volumes this is invisible. Under a marketing push that lands five concurrent checkouts on the same distributor, the third transaction queues, the fourth times out at ten seconds, and the customer sees a failed checkout that may have actually charged their card.

Break scenario
Distributor processes >1 order/sec sustained.
Typical fix
Drop the counter for a timestamp + random ID, or shard into N sibling counters.
02 Critical · silent failure

The weekly digest fetches every order, forever

src/app/api/cron/weekly-digest/route.ts:34–57

For each distributor that opted into the Monday report, the cron issues a single collection('orders').where('distributorId', '==', id).get() — unbounded, no pagination, no limit.

A distributor that grows past ten thousand lifetime orders will silently stop receiving their weekly report. The function times out or runs out of memory, logs an error that nobody reads, and moves to the next tenant. Worst class of failure: correct-looking system, missing behavior, no alert.

Break scenario
Any single distributor with >10k orders.
Typical fix
Paginate by date range — only last 8 days of orders are needed for the digest anyway.
03 Critical · client memory

The inventory page holds the whole catalog in RAM

src/app/(app)/inventory/page.tsx

getAllInventoryRecords loads every record for the distributor into a single JavaScript array on the client. For a small shop this is nothing — a few hundred kilobytes. For a distributor with ten thousand records, it is eight to ten seconds of wait and a browser tab that consumes more memory than the OS allocates to Safari on iPhone.

The symptom is indistinguishable from a bad laptop. Users blame the browser. The page still works, until it doesn't.

Break scenario
Distributor catalog exceeds ~10k records; mobile breaks first.
Typical fix
Move pagination and search server-side; render virtual rows only.

What to fix, and when

Eight moves, ordered by the growth they unblock

  1. Done ✓
    Replace the order counter with a timestamp+random ID
    Shipped · new format PREFIX-YYYYMMDD-XXXXXX. No shared counter, no contention.
    lib/order-number.ts
  2. Do before 10×
    Paginate the weekly digest by date range
    Only last 8 days needed — cap the per-distributor query so it works at any tenant size.
    where('paidAt', '>=', 7d)
  3. Do before 10×
    Server-paginate the inventory list
    Cursor-based fetch + search pushed to Firestore. Client stays lean on any catalog size.
    .limit(50) + .startAfter()
  4. Do before 100×
    Add composite index + per-tenant chunking to payment reminders
    Index on (status, approvedAt); iterate per distributor instead of global scan.
    firestore.indexes.json
  5. Do before 100×
    Move the rate limiter to Upstash or Redis
    Global, cross-instance limits — what the in-memory Map pretends to do but can't.
    Upstash Ratelimit
  6. Do before 100×
    Queue invoice-PDF generation behind Cloud Tasks
    Decouples heavy jobs from the request path. Fan-out stops blowing up Cloud Run memory.
    Cloud Tasks + worker
  7. Do before 100×
    Move Gemini to paid tier with per-distributor quotas
    Removes the 15 RPM ceiling; bills only active tenants; caches repeated bio generations.
    billing + per-tenant meter
  8. Do before 1000×
    Batch per-item stock settlement into a multi-record transaction
    Collapses N serial transactions into a single 500-op bulk write for large orders.
    bulkWriter or single tx
